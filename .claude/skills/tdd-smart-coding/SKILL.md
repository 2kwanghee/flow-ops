# TDD Smart Coding - Red-Green-Refactor 루프

테스트 주도 개발(TDD) 방식으로 기능을 구현합니다. 기능을 최소 단위로 분해하여 Red → Green → Refactor → Commit 사이클을 반복합니다.

## 프로젝트 설정 자동 감지 (시작 시 1회)

구현에 앞서 프로젝트 환경을 파악한다:

1. **패키지 매니저 & 테스트 프레임워크 감지**:
   - `package.json` → npm/yarn/pnpm, jest/vitest/mocha 확인
   - `pyproject.toml` / `setup.cfg` → pytest/unittest 확인
   - `go.mod` → go test 사용
   - `Cargo.toml` → cargo test 사용
2. **린터 & 포매터 감지**: 프로젝트에 설정된 도구 확인 (eslint, prettier, ruff, golangci-lint 등)
3. **기존 커밋 컨벤션 확인**: `git log --oneline -10`으로 프로젝트의 커밋 메시지 스타일을 파악하여 따름

## 워크플로우: TDD 사이클

`$ARGUMENTS`의 작업을 최소 단위로 분해하여 각 단위마다 아래 사이클을 반복한다.

### Step 1: 계획 & 분해
- 작업을 독립적으로 테스트 가능한 최소 단위로 분해
- TODO.md에 분해 결과를 기록하고 현재 진행 단위 표시

### Step 2: 테스트 작성 (Red)
- 현재 단위에 대한 실패하는 테스트를 먼저 작성
- 테스트를 실행하여 실패를 확인 (Red 상태)
- 테스트가 이미 통과한다면: 테스트가 올바른 동작을 검증하는지 재확인

### Step 3: 최소 구현 (Green)
- 테스트를 통과시키기 위한 **최소한의 코드**만 작성
- 필요 이상의 코드를 미리 작성하지 않음 — 테스트가 요구하는 만큼만 구현
- 테스트를 다시 실행하여 통과 확인

### Step 4: 리팩터링 (Refactor)
- 테스트가 통과하는 상태에서 코드 품질을 개선
- 중복 제거, 네이밍 개선, 구조 정리
- 리팩터링 후 테스트를 다시 실행하여 여전히 통과하는지 확인
- 리팩터링할 것이 없으면 이 단계를 건너뛰어도 됨

### Step 5: 린트 & 포맷
- 프로젝트에 설정된 린터/포매터를 실행
- 린트 에러가 있으면 수정 후 테스트 재실행

### Step 6: 커밋
- 관련 파일만 선택적으로 stage (`git add .` 지양)
- 프로젝트의 기존 커밋 컨벤션을 따르는 메시지로 커밋
  - 기존 컨벤션이 없으면: `feat: <설명>`, `fix: <설명>`, `test: <설명>` 형식 사용

### Step 7: TODO 업데이트 & 반복
- TODO.md에서 완료된 단위를 체크
- 다음 단위로 이동하여 Step 2부터 반복

## 루프 시각화

```
┌──────────────────────────────────────────────┐
│  1. 테스트 작성 → 실행 → 실패 확인 (Red)      │
│  2. 최소 구현 → 테스트 통과 (Green)           │
│  3. 리팩터링 → 테스트 여전히 통과 (Refactor)  │
│  4. 린트 & 포맷 → 깨끗한 코드                 │
│  5. 커밋 (작고 집중된 단위)                    │
│  6. 다음 단위 → Step 1로                      │
└──────────────────────────────────────────────┘
```

## 원칙

- **테스트를 먼저 작성하는 이유**: 구현 전에 요구사항을 명확히 정의하고, 구현이 올바른지 즉시 검증할 수 있다. 테스트 없이 구현하면 나중에 검증이 어려워진다.
- **한 사이클에 하나의 기능만 다루는 이유**: 여러 기능을 동시에 구현하면 어떤 변경이 문제를 일으켰는지 파악하기 어렵다. 작은 단위로 나누면 디버깅이 쉽고 되돌리기도 간단하다.
- **테스트가 실패하는 상태에서 커밋하지 않는 이유**: 팀원이나 CI가 항상 green 상태의 코드를 기대한다. 실패하는 커밋은 bisect나 rollback을 방해한다.
- **린트 에러가 있는 상태에서 커밋하지 않는 이유**: 린트 에러는 잠재적 버그이거나 팀 컨벤션 위반이다. CI에서 잡히기 전에 로컬에서 해결한다.
- 테스트 작성이 어렵다면 기능 단위가 너무 큰 것이다 — 더 작게 분해한다.
- 디버깅이 2회 시도 내에 해결되지 않으면 마지막 green 상태로 되돌리고 더 작은 단계로 재시도한다.
- 각 사이클 완료 시 사용자에게 간단한 진행 상황을 보고한다.
